## 2. 状态机 v2.0

### 2.1 6 态枚举

```python
# src/asset_hub/models/asset.py
class AssetStatus(StrEnum):
    IDLE = "IDLE"             # 闲置（v1.0：闲置中）
    IN_USE = "IN_USE"         # 在用（v1.0：使用中）
    MAINTENANCE = "MAINTENANCE"  # 送修（v1.0：维修中）
    BROKEN = "BROKEN"         # 故障（v2.0 新）
    RETIRED = "RETIRED"       # 退役（v1.0：已退役）
    DISPOSED = "DISPOSED"     # 注销（v1.0：已处置）
```

中文文案全 6 态两字对齐。`DISPOSED` 文案从"已处置"改"注销"是为了精确表达"卖/捐/扔/销毁"等账面消除场景。

### 2.2 12 transition kind

```python
# src/asset_hub/models/state_transition.py
class TransitionKind(StrEnum):
    CHECKOUT_INTERNAL = "CHECKOUT_INTERNAL"        # 派发
    CHECKOUT_EXTERNAL = "CHECKOUT_EXTERNAL"        # 出借
    RETURN = "RETURN"                              # 归还
    SEND_TO_MAINTENANCE = "SEND_TO_MAINTENANCE"    # 送修
    RECOVER_FROM_MAINTENANCE = "RECOVER_FROM_MAINTENANCE"  # 维修完成
    RETIRE = "RETIRE"                              # 退役
    REINSTATE = "REINSTATE"                        # 重新启用
    DISPOSE = "DISPOSE"                            # 注销（v1.0 label "处置"）
    REASSIGN = "REASSIGN"                          # 重新分配（v2.0 新，合并 RELOCATE + TRANSFER_HOLDER）
    REPORT_BROKEN = "REPORT_BROKEN"                # 出现故障（v2.0 新）
    DECLARE_UNREPAIRABLE = "DECLARE_UNREPAIRABLE"  # 故障报废（v2.0 新）
    DISMISS = "DISMISS"                            # 故障解除（v2.0 新）
```

v1.0 删除：`RELOCATE` / `TRANSFER_HOLDER`（功能合并到 `REASSIGN`）。

### 2.3 holder/location rule 体系（5 类 enum）

```python
# src/asset_hub/services/state_machine.py
HolderRule = Literal["required", "optional", "forced_null", "ignored", "keep"]
LocationRule = Literal["required", "optional", "forced_null", "keep"]
```

| rule | 语义 | 不传时 | 传值时 |
|---|---|---|---|
| `required` | 必须产生变化动作 | 报错 | 用入参 |
| `ignored` | 必须不变 | 保留 asset 当前值 | 忽略入参 |
| `optional` | 可选留空，默认变空 | 清空（None）| 用入参 |
| `keep` | **可选变化，默认不变**（v2.0 新引入）| 保留 asset 当前值 | 用入参（可空可值）|
| `forced_null` | 必须清空 | 强制 None | 忽略入参 |

`keep` 通过 `_Unset` 哨兵实现（区分 caller "未传字段" vs "显式传 None 清空"），与项目已有 `AssetService.update_asset` 同模式。

**v2.0 不再有 kind 使用 `ignored`**（v1.0 唯一用 ignored 的 RELOCATE 已并入 REASSIGN）。`ignored` 值保留供未来用。

| # | kind | 中文 | valid_from | to | holder | location |
|---|---|---|---|---|---|---|
| 1 | CHECKOUT_INTERNAL | 派发 | {IDLE} | IN_USE | required | keep |
| 2 | CHECKOUT_EXTERNAL | 出借 | {IDLE} | IN_USE | required | keep |

### 2.5 派出集 closes 通用化

v1.0 把"找最近 OPEN CHECKOUT 自动 closes"写死在 `RETURN` 一个 kind 上。v2.0 通用化为"派出集进出检测"：

```python
PERSISTED_CHECKOUT_STATES = frozenset({AssetStatus.IN_USE, AssetStatus.BROKEN})

# transition.py 内
new_status = to_status if to_status is not None else asset.status

closes_id = None
if (asset.status in PERSISTED_CHECKOUT_STATES
    and new_status not in PERSISTED_CHECKOUT_STATES):
    closes_id = self.repo.find_open_checkout_id(asset_id)
    # RETURN 强约束：必须有 OPEN CHECKOUT（v1.0 行为保留）
    if kind == TransitionKind.RETURN and closes_id is None:
        raise IllegalTransitionError(f"资产无未归还的派发记录: {asset_id}")
    # 其他从派出集走出的 transition：closes_id 为 None 也合法
```

`BROKEN` 视为"派出延续状态"——`IN_USE → BROKEN` 时 CHECKOUT 不闭合（持有人/位置仍可能由原派出人保留）；从 `BROKEN` 走出派出集（DISMISS / SEND_TO_MAINTENANCE / RETIRE / DISPOSE）才闭合。

`REASSIGN` 因 `to_status=None`，`new_status == asset.status`，永远不触发 closes（资产仍在原态）。

### 2.6 特殊约束清单

| kind | 约束 |
|---|---|
| `CHECKOUT_*` | 写入时 record OPEN，等闭合 |
| `RETURN` | 自动 closes 最近 OPEN CHECKOUT；找不到报错（保 v1.0） |
| `SEND_TO_MAINTENANCE` | 仍禁止 `IN_USE` 直跳（先 RETURN 或 REPORT_BROKEN）；`BROKEN → MAINTENANCE` 新允许 |
| `RETIRE` | CLI `--yes` / `--dry-run` + GUI AlertDialog |
| `DISPOSE` | CLI `--yes` / `--dry-run` + GUI confirm phrase **`"注销"`**（v1.0 `"处置"` 改名） |
| `REASSIGN` | service 校验：必须改 holder 或 location 至少一项；CLI / dialog 同步实施 |
| `REPORT_BROKEN` | `IN_USE → BROKEN` 时**不**闭合 OPEN CHECKOUT（派出延续语义，靠通用化逻辑自动判断） |
| `DECLARE_UNREPAIRABLE` | CLI `--yes` / `--dry-run` + GUI AlertDialog（与 RETIRE 同级） |
| `DISMISS` | 无特殊；走通用化 closes 逻辑 |

| 字段 | 类型 | 用途 |

#### Schema 层

- 历史 RELOCATE/TRANSFER_HOLDER 记录被 migration 改写为 REASSIGN（作者无正式录入数据，单向不可逆）

#### HTTP API 层

- `POST /api/assets/{id}/transitions` body.kind 旧值（`RELOCATE` / `TRANSFER_HOLDER`）现 422
- body 字段 `to_holder` / `to_location` 语义微变（部分 kind 现 `keep` 而非 `optional`；"未传 vs 传 null" 区分）
- envelope error 多 `hint` / `fields_missing` 等可选字段（向后兼容增）

#### CLI 层

- 删 `asset relocate` / `asset transfer-holder` 子命令
- 加 4 个新子命令（见 §3.4.1）
- 12 处 flag rename（见 §3.4.2）
- `asset dispose` 交互 prompt `"处置"` → `"注销"`
- 增 `--help --json` 双模、`--fields` 字段掩码（向后兼容增）

#### UI / 文案层

- 6 态中文文案全改（含已有 5 态从三字 → 两字）
- `DISPOSE` UI label / KIND_META label "处置" → "注销"
- GUI dispose-alert-dialog confirm phrase "处置" → "注销"
- 导出 CSV/XLSX 列值变（旧脚本按字符串匹配会断）

### 5.3 SemVer 判定：v2.0.0

含 HTTP API 契约破坏 + CLI 子命令删除 + flag rename 12 处 + UI 文案 breaking，严格按 SemVer 标 MAJOR。`v2.0.0`。

### 5.4 风险与回滚

#### 主要风险

| 风险 | 缓解 |
|---|---|
| migration 数据迁移失败 | 作者无正式录入数据；测试库验证；备份强制 |
| CLI flag breaking 漏改文档 | SKILL.md / references 改动是 PR-1 一部分，CI lint 强校验 |
| description optimization 改丢 v2.0 语义 | eval baseline 用 PR-1 改完版本；loop 仅改触发关键词不改语义 |
| 前端 generated schema 漂移 | 后端 schema 改后强制 `pnpm gen:api`；CI schema diff check（独立 followup）|
| breaking 漏列 | spec 自查阶段对照 §5.2 完整复核 |

#### 回滚

```bash
git checkout v1.0.0
cp data/asset_hub.db.v1.0.bak data/asset_hub.db
uv sync && pnpm --dir frontend install
uv run asset-hub serve restart --mode prod
```

#### 不可回滚

- v2 期间产生的 BROKEN 状态资产 / REPORT_BROKEN / DECLARE_UNREPAIRABLE / DISMISS / REASSIGN transition records
