## 0. 摘要

向 `Asset` 模型新增 `model: str \| None` 顶层字段（nullable，非 unique），与 `serial_number` 同结构等级；同步打通搜索、排序、导出、CLI flag、前端表单/详情/列表的覆盖。顺修 v1 遗留 bug：`SortByField` 加入 `serial_number`（前端可点 sort 但后端不响应的不一致）。删 `examples/types/gpu.json` 中冗余的 `model` custom_field。

字段语义**不强约束**：用户/Agent 可按场景自由填（产品系列名、精确型号、品牌+型号等）。表单、SKILL.md 不为 `model` 加专属 helper text / Gotcha / 示例段，全部与其他字段（特别是 `serial_number`）同等对待。

## 1. 目标 & 范围

### 1.1 解决的问题

- `Asset.name` 单字段语义过载：既装"配给李四的开发本"也装"ThinkPad X1 Carbon"
- 跨 type 的"型号"信息没有统一存储位置，列表/搜索/导出做不出一致呈现
- `SortByField` 不含 `serial_number`：前端表格 `serial_number` 列有排序 UI 但后端 API 不接，是 v1 遗留小不一致

### 1.2 范围

- 数据模型 + alembic migration（v3）
- Service / API DTO / router
- CLI register / update / list
- 前端表单 / 详情 header / 详情 general-fields / 列表表格 / column-visibility / search schema / export
- examples/types 清理
- SKILL.md 字段表更新

### 1.3 不做

- 不强约束 `name` / `model` 语义（不在表单加 helper text、不在 SKILL.md 加专属 Gotcha 或示例段）
- 不 backfill 现有 `name` 数据（不尝试 split 历史 name 为 name + model）
- 不动 `holder` / `location` / `notes` / `custom_data` 任何处理逻辑
- 不为 `model` 加 unique 约束
- 不新增 e2e 专属 spec（仅现有 spec 加最小列断言）

## 2. 数据模型

### 2.1 字段定义

```python
# src/asset_hub/models/asset.py
class Asset(SQLModel, table=True):
    __tablename__ = "assets"

    id: uuid.UUID = Field(default_factory=uuid.uuid4, primary_key=True)
    asset_code: str = Field(unique=True, index=True)
    serial_number: str | None = Field(default=None, unique=True, index=True)
    name: str = Field(index=True)
    model: str | None = Field(default=None, index=True)  # 新；非 unique，加 index 与 name 对齐
    type_id: uuid.UUID = Field(foreign_key="asset_types.id", index=True)
    status: AssetStatus = Field(default=AssetStatus.IDLE)
    # ... 其余字段不变
```

**位置**：紧贴 `name` 之后、`type_id` 之前。理由：`name` 与 `model` 同属"描述性字段"，相邻更直观。

**索引决策**：加 `index=True`，与 `name` 字段对齐——二者同档同处理。即便模糊搜索 LIKE '%foo%' 用不上 B-tree 索引，排序场景（`?sort=model` / `?sort=-model`）会用到。不加 unique（多台同型号资产正常）。

### 2.2 alembic migration（v3）

新建 migration 文件：`src/asset_hub/alembic/versions/20260513_<hash>_v3_asset_model_column.py`

```python
def upgrade() -> None:
    with op.batch_alter_table("assets") as batch_op:
        batch_op.add_column(sa.Column("model", sa.String(), nullable=True))
        batch_op.create_index("ix_assets_model", ["model"])


def downgrade() -> None:
    with op.batch_alter_table("assets") as batch_op:
        batch_op.drop_index("ix_assets_model")
        batch_op.drop_column("model")
```

- 使用 `render_as_batch=True`（项目 alembic env 已配置，SQLite 改表必需）
- nullable=True，不需要 server_default
- 不 backfill 现有 `name` 值——历史 name 保持原样

`down_revision` 指向 v2 state machine migration（PR-1 落地的 head）。

### 2.3 顺修 v1 遗留 bug：SortByField

`src/asset_hub/services/asset.py`：

```python
# 旧
SortByField = Literal["name", "asset_code", "created_at", "updated_at", "acquired_at", "idle_days"]
SORT_FIELDS = ("name", "asset_code", "created_at", "updated_at", "acquired_at", "idle_days")

# 新（加 model / serial_number）
SortByField = Literal[
    "name", "model", "asset_code", "serial_number",
    "created_at", "updated_at", "acquired_at", "idle_days",
]
SORT_FIELDS = (
    "name", "model", "asset_code", "serial_number",
    "created_at", "updated_at", "acquired_at", "idle_days",
)
```

**理由**：
- 前端 `assets-table.tsx` 当前对 `serial_number` 列允许 sort（无 `enableSorting: false`），但 backend `SortByField` 不含——用户点 sn 列 header 时，URL `?sort=serial_number` 传给后端会被拒/忽略。PR-3 同时引入 model 列与排序，必须正确实现，顺手把 sn 修齐。

### 4.2 `asset update`

```python
@app.command("update")
def update(
    asset_id: str = typer.Argument(...),
    name: Annotated[str | None, typer.Option("--name")] = None,
    model: Annotated[str | None, typer.Option("--model")] = None,  # 新
    serial_number: Annotated[str | None, typer.Option("--serial-number")] = None,
    # ...
):
    # 用 parse_unset_or_value 区分"未传 / 显式清空 / 新值"
    payload = {}
    if model is not None:
        payload["model"] = parse_unset_or_value(model)
    # ...
```

`parse_unset_or_value` 约定（PR-1 引入）：
- Typer 默认 None → 未传，service 收 `_UNSET`，保持现状
- 空字符串 `""` → 显式清空，service 收 `None`
- 非空字符串 → 新值

`asset list` 表格输出列：DTO 字段全展示——current 行为已含 name + sn 等，自动获得 model 列。具体表格列宽 / 截断由 `services/asset.py` 内 ASCII table 渲染逻辑决定（沿用现状）。

## 5. 前端

### 5.1 表单 `general-fields-form.tsx`

在 `name` 字段之后、`type_id` 字段之前插入 `model` 字段：

```tsx
<FormField
  control={control}
  name={modelField}
  render={({ field }) => (
    <FormItem>
      <FormLabel>型号</FormLabel>
      <FormControl>
        <Input
          {...field}
          value={field.value ?? ''}
          placeholder="如 ThinkPad X1 Carbon Gen 9（可空）"
        />
      </FormControl>
      <FormMessage />
    </FormItem>
  )}
/>
```

**对齐细节**：
- **无** `*` 必填标记（与 sn、holder、location、notes 同）
- **无** `font-code` className（model 是产品名，非机器码——与 name 同字体）
- **无** `FormDescription` helper text（无字段单独突出）
- placeholder 风格短描述（与 sn `"厂家铭牌编号（可空）"` 同长度等级）

表单字段最终顺序：

```
[assetCode readonly] → name * → model → type * → sn → acquired_at → [holder] → [location] → notes
```

### 5.2 详情 header `asset-header.tsx`

参照 `type_name` 的双重出现模式（header 副行 + general-fields 行都出现），把 `asset_code · type_name · StatusBadge` 那行扩展：

```tsx
<div className="flex items-center gap-3">
  <span className="font-code text-sm text-muted-foreground">
    {asset.asset_code}
  </span>
  <span className="text-sm text-muted-foreground">·</span>
  <span className="text-sm text-muted-foreground">
    {asset.type_name ?? "未知类型"}
  </span>
  {asset.model && (
    <>
      <span className="text-sm text-muted-foreground">·</span>
      <span className="text-sm text-muted-foreground">{asset.model}</span>
    </>
  )}
  <StatusBadge status={asset.status} />
</div>
```

**渲染规则**：
- model 为 null/空时整段（`· model`）不渲染——避免视觉空洞与 "—" 占位的噪音
- 字号 / 颜色与 `type_name` 完全一致（`text-sm text-muted-foreground`）

### 5.3 详情 general-fields `general-fields.tsx`

在 `"类型"` 行之后、`"当前持有人"` 之前插入 `"型号"` 行：

## 8. 升级路径 & breaking changes

### 8.1 升级路径

```bash

uv run alembic upgrade head  # 跑 v3 migration（加 model 列）
```

### 8.2 breaking changes

#### Schema 层（非 breaking，addition）

- Asset 新增 nullable `model` 字段
- 历史数据 `model = NULL`，行为零变化

#### HTTP API 层（非 breaking，addition）

- `POST /api/assets` body 接受可选 `model`
- `PATCH /api/assets/{id}` body 接受可选 `model`
- `GET /api/assets` 响应含 `model` 字段
- `GET /api/assets?q=foo` 搜索范围扩到 model + sn（行为变化但向后兼容——查得更全不少结果）
- `GET /api/assets?sort=model` / `?sort=serial_number` 现可用

#### CLI 层（非 breaking，addition）

- `asset register` / `asset update` 加 `--model` flag
- `asset list --sort model` / `--sort serial_number` 可用

#### v1 顺修：SortByField 行为变化

- v1 前端可点 sn 列 sort，URL 传 `sort=serial_number` 给后端被忽略——v2.0 PR-3 后正确响应
- 用户视角：以前点 sn header 列表"看起来"按 sn 排了（前端本地排序生效），实际 URL 一刷新就掉回默认排序；现在 URL 持久化正常
- 这是修 bug，不算 breaking

### 8.3 SemVer 判定

PR-3 单独看是 MINOR addition；但本身作为 v2.0.0 的一部分发出，跟随 PR-1 的 MAJOR 标签——不单独 bump。

### 8.4 风险

| 风险 | 缓解 |
|---|---|
| migration 跑失败 | 单文件加列 + nullable，几乎不可能失败；测试库验证 + 备份强制 |
| 前端 generated schema 漏拉 | CLAUDE.md 已写明强制 `pnpm gen:api`，CI 可补 schema diff check（独立 followup） |
| 搜索扩展引入 N+1 | 三列 OR ilike 在 < 10k 数据量下可接受；后续若慢可加全文索引 |
| 已存在的 GPU AssetType（按 v1 example 创建）仍含 model custom_field | example 文件改动**不**自动迁移已存在的 type definition。如用户曾按 v1 `gpu.json` 创建过 GPU type，其 DB 中 `asset_types.custom_fields` 仍含 `{"key": "model", ...}` 项——新创建 GPU 资产时录入表单会出现"顶层 model 字段 + custom model 字段"双重要求。**解决**：release-notes-v2.0 提示用户在 type 管理页面手动编辑 GPU type 删除该 custom_field；不写 data migration（单用户工具，手工迁更简单可靠）。已录入资产的 `custom_data` 中的 `model` 键保留不影响（JSON 弹性结构） |

PR-3 严格串行在 PR-2 之后：

### 9.5 串行策略 vs 并行的取舍

并行（PR-2 / PR-3 同时开发）的冲突面（已分析）：
- `schema.d.ts` 必冲突（双方生成）
- `release-notes-v2.0.md` 必冲突（双方 new file）
- `SKILL.md` / `cli/asset_cmd.py` 概率冲突（不同段加新内容）

冲突量级是"5-10 分钟手工 rebase"——技术上可行。但**选择串行**的理由：
- PR-3 工作量预估比 PR-2 剩余工作量小（PR-3 单纯加字段；PR-2 剩 Phase 4-6 含 `--fields` 基础设施 + description eval + 收尾）——等 PR-2 完无需太久
- 串行 = 零冲突 = 零 rebase 心智成本
- PR-3 plan 阶段可基于 PR-2 落地后的完整图景写（如能借用 PR-2 加的 envelope `fields_invalid` 来报告 model `--fields` 未知字段错误，更一致）
