## 0. 摘要

新增 `.github/workflows/ci.yml`，并行跑后端（`ruff check` + `pytest`）与前端（`pnpm lint` + `tsc -b` + `vitest run`）两条 job，pull_request 与 push main 双触发，10 分钟 timeout。

**有意不做**：
- 不加 `ruff format --check`：项目 80/142 个 Python 文件未按 ruff format 规范格式化（早期埋的死契约），引入即 fail。`continue-on-error` 是 anti-pattern（让人忽略红 X）。该步骤改由 **PR-B（ruff format baseline）** 在 reformat 全部文件的同一 PR 内一次性接入，逻辑闭环更干净
- 不动 `.github/workflows/e2e.yml`：e2e 独立 gate（playwright 浏览器装机慢、artifact 上传逻辑独立），合并反而降低可维护性
- 不上 Windows runner：成本/时长 ~2-3x，且本仓库无 Windows-only 代码路径（serve start 的进程树管理 psutil 跨平台），开发者本机日常即 Windows 手测
- 不引入 prettier：用户原意已确认（80+ 文件 reformat 噪音 + git blame 失效；当前 ESLint 规则集合够用）

## 1. 目标 & 范围

### 1.1 解决的问题

现有 CI（`e2e.yml`）只跑前端 Playwright chromium 一条 lane，609 后端测 + 179 前端 vitest 测 + 后端 ruff lint + 前端 ESLint + 前端独立 `tsc -b` type check 全部依赖维护者本地"我跑过了"。v2.0 GA 后进入 polish 阶段，需先把"机器视角的最低质量门"立住，后续 PR-B/PR-C 才能在 regression 自动可见的前提下推进。

### 1.2 范围

- 新建 `.github/workflows/ci.yml`，2 个 job：`backend` / `frontend` 并行
- 与现有 `e2e.yml` 触发条件保持一致（`pull_request: branches: [main]` + `push: branches: [main]`）
- 缓存：setup-uv 自带 `enable-cache: true`，setup-node `cache: pnpm` + `cache-dependency-path`
- timeout：每 job 10 分钟
- runner：`ubuntu-latest`

### 1.3 不做

- 不接入 `ruff format --check`（理由见 §0）
- 不引入 prettier（理由见 §0）
- 不动 `e2e.yml`（理由见 §0）
- 不加 Windows runner
- 不引入 codecov / coverage 上传（YAGNI，v2.x polish 不在范围内）
- 不引入 concurrency cancel（小仓库 / 单维护者，价值低；后续撞到再加）
- 不引入 matrix（Python/Node 单版本足够；后续要双版本再加）

## 2. 文件与结构

### 2.1 新增 `.github/workflows/ci.yml`

```yaml
name: ci

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  backend:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v3
        with:
          version: "0.5.x"
          enable-cache: true
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: uv sync
      - name: ruff check
        run: uv run ruff check .
      - name: pytest
        run: uv run pytest

  frontend:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    defaults:
      run:
        working-directory: frontend
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 10
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
          cache-dependency-path: frontend/pnpm-lock.yaml
      - run: pnpm install --frozen-lockfile
      - name: ESLint
        run: pnpm lint
      - name: tsc -b
        run: pnpm exec tsc -b
      - name: vitest
        run: pnpm test
```

### 2.2 与 `e2e.yml` 的关系

`e2e.yml` 保持原样不动。`ci.yml` 与 `e2e.yml` 共享触发条件——一次 push / PR 同时启动两个 workflow，**完全并行**（GitHub Actions 平台行为，无需显式配置）。三个 job 总计：

| workflow | job | 跑什么 |
|---|---|---|
| ci | backend | ruff check + pytest |
| ci | frontend | ESLint + tsc -b + vitest |
| e2e | e2e | playwright chromium |

## 3. 关键决策

### 3.1 单 workflow 含 2 job 还是 2 workflow？

选**单 workflow 2 job**。

理由：
- 后端/前端在概念上同属"日常 CI gate"，逻辑分组清晰
- GitHub UI 中 workflow 是顶层分组，一个 `ci` workflow 内并列两个 job 在 PR check list 显示为 `ci / backend` / `ci / frontend`，比拆两个独立 workflow 视觉更紧凑
- 未来若加 lint-only 或 format-only job 也归到 `ci` 自然

### 3.2 不上 matrix（Python 3.12 / Node 20 单版本）

`pyproject.toml` 明写 `requires-python = ">=3.12"`，但生产只跑 3.12；Node 20 是 LTS。加 matrix 会让 CI 时长翻倍但 catch 不到额外 bug——后续真要测多版本兼容再加。

### 3.3 不加 `concurrency.cancel-in-progress`

单维护者 + PR 频率低（v2.0 阶段 ~3-5 PR/周），冲撞队列概率极低；加了反而在批量 push fix-up commit 时频繁取消刚开始的 run 浪费 cache warm-up。

### 3.4 `frontend` job 用 `defaults.run.working-directory`

避免每步重写 `working-directory: frontend`。`pnpm/action-setup` 与 `actions/setup-node` 不属于 `run` step，不受影响。

### 3.5 `tsc -b` 而非 `tsc --noEmit`

遵守 user memory feedback：`pnpm --dir frontend build` 跑的就是 `tsc -b && vite build`；单 pass `--noEmit` 会漏 7+ 真错。`tsc -b` 走 project references，独立于 vite build 提早暴露类型错。

### 3.6 `pnpm install --frozen-lockfile` 而非 `pnpm install`

严格按 lockfile 装，CI 上 lockfile 漂移立刻 fail——与 e2e.yml 一致。

## 4. 触发面与失败行为

| 事件 | ci.yml | e2e.yml |
|---|---|---|
| PR open → main | ✅ | ✅ |
| PR sync（push to PR branch）| ✅ | ✅ |
| push main（含 merge）| ✅ | ✅ |
| 其他分支 push | ❌ | ❌ |

任一 job fail → workflow fail → PR check 红 X → 维护者必须修。

## 5. 验证策略

### 5.1 本地预跑

实施 worktree 内先按 ci.yml 的命令本地跑一遍（顺序：`uv sync` → `uv run ruff check .` → `uv run pytest` → `cd frontend && pnpm install --frozen-lockfile` → `pnpm lint` → `pnpm exec tsc -b` → `pnpm test`）。若任一步 fail，先在本地修透——CI 不是发现 bug 的地方，是守住 bug 的地方。

### 5.2 PR 实战验证

推 PR-A 后，依赖 GitHub Actions 实际跑过 backend + frontend + e2e 三 job 全绿才 merge。若 backend 因为本地未跑过某测出现 platform-specific 失败（如 Windows 路径 vs Linux），现场修。

---

## 6. 风险与回退

### 6.1 主要风险

- **CI cache miss 首跑慢**：首次 uv / pnpm cache 冷启可能 backend job 跑 4-5 分钟、frontend 5-6 分钟。10 分钟 timeout 留有 ~2x 余量，可接受
- **现有代码 lint/test 实际不绿**：极不可能（v2.0 GA 前维护者本地跑过 609 测 + ruff check 全绿），但若 CI 跑出意外失败，按"本地反推"原则现场修而非降低门槛
- **frontend `tsc -b` 真有错**：可能性低（pnpm build 在 v2.0 阶段反复跑过），若有先 inspect 再决定是修还是临时降级

### 6.2 回退

ci.yml 是新文件，回退即 `git revert` 一个 commit。不影响 e2e.yml、不影响代码。

## 7. 后续 PR 的接口

- **PR-B（ruff format baseline）**：在 80 文件 reformat 的同一 PR 内向 `ci.yml` `backend` job 追加 `uv run ruff format --check .` 步骤（紧接 `ruff check` 之后），保证从那一刻起 PR 自动守护格式
- **PR-C（serve port detection）**：纯 runtime 修复，不动 CI；但 PR-A 合入后，PR-C 的 backend 改动会自动被 `pytest` + `ruff check` 守护
