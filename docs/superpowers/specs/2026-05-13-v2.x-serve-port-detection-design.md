## 0. 摘要

修 v1/v2 起就有的隐藏 bug：`asset-hub serve start --mode dev` 不识别外部 Vite 进程占用的 `5173`，启动后 Vite 自动 fallback 到 `5174` 但 PID 文件 + `status` 仍报 `frontend.port: 5173`，导致用户访问 5173 看到旧 bundle、访问 5174 才是新代码。

**根因（v2.0 spec 文档第一次明确记录）**：

- `src/asset_hub/cli/serve/proc.py::is_port_in_use(port, host="127.0.0.1")` 只测 IPv4
- Vite 8.x dev 默认监听 **IPv6 `::1`**（见 `lifecycle.py:144-146` 已存在的注释 "Vite 8.x dev 默认只绑 IPv6 ::1，IPv4 127.0.0.1 会被拒绝"）
- 外部 Vite 占住 `::1:5173` 时本机 `127.0.0.1:5173` 仍可 bind → 漏检
- start_service 不 raise → 启 Vite → Vite 内部检测 `::1:5173` 已占 → 偷偷 fallback 5174 → PID 文件记 5173 撒谎

**修复（最小有效切片 (a) + (a')）**：

1. **(a)** `proc.is_port_in_use` 升级为双栈探测：IPv4 `127.0.0.1` + IPv6 `::1` 任一被占即返 True
2. **(a')** `frontend/vite.config.ts` 加 `server.strictPort: true` 兜底：万一 (a) 漏，Vite 也直接 fail 而非 fallback，让 start_service Phase 4 frontend probe 失败 → rollback + clean error

**有意不做 (b)（doctor 加"端口占用者 PID 不在 serve 管理范围"检测）**：(a) 升级 `is_port_in_use` 后 `doctor.check_port_free` 自动得到双栈覆盖；额外的 "occupier PID 是否在 serve 管理范围" 检测属于 doctor 噪音强化，本 bug 不需要。

**有意不做 (c)（从 Vite stderr 解析实际端口写 PID 文件）**：(a)+(a') 让 fallback 路径不存在，PID 文件中"设定端口"就是"实际监听端口"。加 port 字段反而引入 plain-text PID 文件 schema 漂移，YAGNI。

## 2. 代码改动

### 2.1 (a) `proc.is_port_in_use` 双栈探测

当前实现（`src/asset_hub/cli/serve/proc.py:85-95`）：

```python
def is_port_in_use(port: int, host: str = "127.0.0.1") -> bool:
    """检查 host:port 是否被占用（true = 已占用）。"""
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.settimeout(0.5)
    try:
        s.bind((host, port))
    except OSError:
        return True
    finally:
        s.close()
    return False
```

新实现：

```python
def is_port_in_use(port: int) -> bool:
    """检查指定端口是否被占用（IPv4 127.0.0.1 或 IPv6 ::1 任一被占即 True）。

    Vite 8.x dev 默认只监听 IPv6 ::1；uvicorn 显式 --host 绑 IPv4 127.0.0.1。
    任一栈被占用都需要识别，否则会出现"IPv4 探测显示空闲但 IPv6 已被占"的漏检，
    导致 Vite 偷偷 fallback 到下一个端口而 PID 文件撒谎。
    """
    for family, host in (
        (socket.AF_INET, "127.0.0.1"),
        (socket.AF_INET6, "::1"),
    ):
        s = socket.socket(family, socket.SOCK_STREAM)
        s.settimeout(0.5)
        try:
            s.bind((host, port))
        except OSError:
            return True
        finally:
            s.close()
    return False
```

**API 破坏**：去掉 `host: str = "127.0.0.1"` 参数（不再有意义；双栈固定）。grep 调用方确认只有 `lifecycle.py` 和 `doctor.py` 调用，且都按默认 host 调用（没传第二参数），所以删除参数无 caller 改动。

**测试 mock 影响**：`tests/cli/test_serve_start.py` / `tests/unit/test_doctor.py` 已经 mock `is_port_in_use` lambda——签名简化不影响 mock 行为（lambda 仍接 `*a, **kw`）。

### 2.2 (a') Vite strictPort

`frontend/vite.config.ts:18-26` 当前：

```typescript
  server: {
    port: 5173,
    proxy: { ... },
  },
```

新增 `strictPort: true`：

```typescript
  server: {
    port: 5173,
    strictPort: true,
    proxy: { ... },
  },
```

**Vite 行为**：`strictPort: true` 时端口被占直接退出（非零 exit code），不再 fallback。这样即使 (a) 在某些奇特网络栈漏检，Vite 也不会撒谎；start_service Phase 4 的 frontend probe (`probe_once("http://localhost:5173/")`) 会失败 → rollback。

### 2.3 测试加固

`tests/unit/test_proc_port_check.py` 现有两测：IPv4 占用 → True；IPv4 free → bool。加两个 IPv6 场景：

- `test_port_in_use_returns_true_for_ipv6_only`：bind `::1:port` 监听后调 `is_port_in_use(port)` 应返 True
- `test_port_in_use_handles_ipv6_disabled_gracefully`：模拟无 IPv6 系统（mock `socket.socket(AF_INET6)` 抛 OSError），应继续走 IPv4 不崩——`is_port_in_use` 仍能返回 bool

## 3. 关键决策

### 3.1 双栈而非"系统所有接口"

不用 `socket.bind(("", port))`（绑 INADDR_ANY），原因：bind ANY 在某些 OS（macOS / Linux）上会失败但 IPv4 / IPv6 单点仍可 bind——反而漏检。逐栈 bind 是最稳的。

### 3.2 不用 connect() 探测

`socket.connect(("localhost", port))` 看连通性是另一种方案，但它依赖端口上服务真在监听 + accept；若进程 listening 但没 accept（如 BACKLOG 满），connect 也可能成功——不可靠。bind 探测是"端口能否拿来用"的直接验证。

### 3.3 删除 `host` 参数（API 破坏）

`is_port_in_use(port, host="127.0.0.1")` 的 `host` 参数现实中**没人传非默认值**（grep `is_port_in_use\(` 仅两处调用，皆默认）。保留参数意味着接口允许"只检测某一栈"，与双栈语义冲突。直接删，调用方零改动。

### 3.4 Vite strictPort vs 命令行 --strictPort

两种方式：
- (i) `vite.config.ts` 设 `server.strictPort: true`（**选这个**）
- (ii) `pnpm --dir frontend dev` 命令加 `--strictPort` 透传

(i) 集中、声明式、与 vite 8.x docs idiom 一致；(ii) 让"strictPort"这一行为分散在 CLI command 字符串里，未来维护者改 lifecycle.py 命令拼接时容易漏。

### 3.5 不引入 port 字段到 PID 文件

(a)+(a') 让"设定 port"与"实际 port"必然一致（不一致则 Vite 直接 fail）。加 port 字段是 over-engineering，会让 PID 文件 schema 漂移（plain-text 3 行 → 4 行），现有 4 个测试文件都要改。

### 3.6 不加 e2e specs / playwright 测

bug 复现需要"两个并行 Vite"才能稳定触发，CI 单 runner 上 5173 干净，e2e 不可能撞到。本地手测能复现的场景，单测 + vite.config.ts 改即可。

## 4. 风险与回退

### 4.1 主要风险

- **IPv6 不可用的极端环境**：某些容器 / VM 关了 IPv6 → `socket.socket(AF_INET6)` 直接抛 OSError。需要 graceful fallback——`test_port_in_use_handles_ipv6_disabled_gracefully` 覆盖。修法：`socket.socket(AF_INET6, ...)` 包 try/except OSError，IPv6 不可用就当作"那一栈不占"继续 IPv4 单栈检查
- **Vite strictPort 影响 e2e**：e2e CI runner 上 5173 干净所以一次起就成功；不会受影响。本地开发者若手动起了 Vite 占 5173 然后又跑 `serve start`，现在会**立即** fail（PR-C 目标行为）而不是 fallback 撒谎——这是改进非 regression
- **生产 prod 模式**：不启 Vite，frontend 由 uvicorn StaticFiles 托管，`is_port_in_use` 升级对 prod 行为无影响

### 4.2 回退

- 仅回退 (a)：`git revert <commit (a)>`，保留 strictPort 兜底
- 仅回退 (a')：`git revert <commit (a')>`，保留双栈 is_port_in_use
- 两 commit 各自独立、互不耦合
