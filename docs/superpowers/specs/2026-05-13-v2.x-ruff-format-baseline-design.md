## 0. 摘要

执行 `uv run ruff format .` 把 80 个未格式化的 Python 文件按当前 ruff 配置一次性 reformat 到 baseline，并在 `.github/workflows/ci.yml` `backend` job 追加 `uv run ruff format --check .` 步骤，让该 baseline 进入 CI 强制守护。

**两个 commit 拆分**：
1. `style(format): ruff format baseline reformat 80 files`（**纯机械改动**，无需逐行审）
2. `ci: backend job 加 ruff format --check`（CI 行为变化，独立可审）

这样合入 main 时 reviewer 看 commit 列表能立刻区分两类改动；`git blame` 上 80 文件的格式化噪音只压在一个 commit 上。

## 1. 目标 & 范围

### 1.1 解决的问题

- `pyproject.toml` 早就声明 `[tool.ruff.lint] ignore = ["E501"]` 加注释"ruff format 负责行长控制"，但 80/142 个文件实际未按 ruff format 格式化——名义契约与实际现状脱节
- PR-A 已上 `ruff check` CI gate，但 `ruff format --check` 缺位 → 任何新代码可以以与 ruff format 不一致的风格混入，技术债持续扩大
- 维护者本地写代码时 IDE 集成的 ruff format / 黑/或者 ruff format on save 与 CI 守护一致，才能真正消除"PR 风格漂移"

### 1.2 范围

- 跑一次 `uv run ruff format .`，对 80 个文件做机械 reformat
- 跑 `uv run pytest` + `uv run ruff check .` 确认 reformat 0 行为变化、不引入 lint regression
- 改 `.github/workflows/ci.yml`：在 `backend` job 的 `ruff check` 步骤后追加 `ruff format --check` 步骤
- 写 spec + plan + 实施

### 1.3 不做

- 不加 pre-commit hook（YAGNI；维护者本地 IDE 集成 + CI `--check` 已构成双层守护）
- 不加 `.editorconfig`（与 ruff format 没本质 overlap，且 ruff format 已自治）
- 不动 `pyproject.toml` `[tool.ruff]` 配置（line-length=88、target-version=py312、ignore E501 全保留——本 PR 是按现有配置 reformat，不是改配置后 reformat）
- 不锁 ruff 版本号（`dev` group 当前 `ruff>=0.8`，实际装 0.15.10；若未来升级带来 format 行为微调，下次 CI `--check` 自动抓到再追加一次小 reformat 即可——不主动锁版本因为锁紧反而拖累常规升级）
- 不动前端 prettier / ESLint stylistic（用户已确认前端不引 prettier）
- 不动 `examples/` 下的 JSON（ruff format 不处理 JSON）
- 不删 `pyproject.toml` `ignore = ["E501"]` 后那行"ruff format 负责行长控制"注释——注释表述仍准确

## 2. 操作步骤

### 2.1 reformat（commit 1）

```bash
uv run ruff format .
```

预期产物：80 个文件改动，diff 全部是空白/换行/参数拆分（无逻辑变化）。机械改动模式：

- 模块 docstring 后强制 blank line
- 多参数函数签名 / 调用从单行展开为每参数一行（当超过 line-length 时）
- `if`/`return` 等控制流 inline 形式按 ruff 规则展开
- 字符串 quote 风格统一（`"` 优先）
- 末尾换行规范

**验证**：

```bash
uv run pytest                  # 应仍 609 passed / 1 skipped
uv run ruff check .            # 应仍 All checks passed
uv run ruff format --check .   # 应输出 "142 files already formatted"，退出 0
```

任何 pytest 失败或 ruff check regression 都是异常——ruff format 是确定性 AST 重排，不该影响语义。失败则停下分析，不强推。

### 2.2 CI 接入（commit 2）

`.github/workflows/ci.yml` `backend` job，在现有 `ruff check` 步骤之后插入：

```yaml
      - name: ruff format check
        run: uv run ruff format --check .
```

理由放在 `ruff check` 之后：lint 失败比格式化失败诊断价值高，先跑 lint 让 reviewer 优先看到那个；但又在 pytest 之前，因为 format 检查比 609 测快得多（< 1s），fail fast。

### 2.3 不需要改的文件

- `pyproject.toml` `[tool.ruff]` 段 0 改动
- 前端任何文件 0 改动
- 文档（除 spec/plan/followup）0 改动

## 3. 关键决策

### 3.1 commit 拆 2 个而不是 1 个

合并成 1 个：reviewer 看到 80 文件 + 1 行 YAML 混在一个 diff，难分辨"哪些是机械、哪些是行为变化"。

拆 2 个：commit 1 message 明确"机械改动，无需逐行审"，commit 2 是 1 行 YAML 行为变化——reviewer 5 秒决策。

### 3.2 不锁 ruff 版本

`dev` group `ruff>=0.8`，当前装 0.15.10。锁严（`ruff==0.15.10`）会拖累后续 `uv sync` 升级。CI `--check` 是动态守护——如果 ruff 升级带 format 行为微调，下次 PR 的 CI 会标红，那时再补一个微 reformat commit 就行，零成本可观察。

### 3.3 不加 pre-commit hook

pre-commit 框架本身要装、配 yaml、做自更新策略，且本仓库是单维护者——本地 IDE 集成 ruff format on save 已经覆盖维护者侧；CI 是终极守护。pre-commit 是有效但 over-engineering。

### 3.4 commit 1 命名 `style(format):` 而非 `chore(format):`

Angular Commit Message Convention 中 `style` 专门用于"不影响代码语义的格式化改动"——本场景的标准类型。`chore` 偏向构建脚本、依赖。

### 3.5 不动 `pyproject.toml` 那条注释

`# ruff format 负责行长控制` 这条注释解释为什么 `E501` 被 ignore——本 PR 落地后该解释**更**准确（之前是空口承诺，现在 CI 真的强制 format）。删了反而丢失上下文。

## 4. 风险与回退

### 4.1 主要风险

- **reformat 引入 pytest 失败**：极不可能（ruff format 是 AST 重排不动语义），但若发生则停下复核——可能是 `# type: ignore` / `# noqa` 注释位置被 reformat 移走、或多行字符串拼接因换行被改了内容。修法是手动 `# fmt: off` / `# fmt: on` 包住那段。
- **80 文件大 diff 让 git blame 失效**：是的，这是 reformat baseline 的必然成本。`.git-blame-ignore-revs` 文件可以用——但 GitHub 自动应用需要 repo settings 启用，且仓库内文件单独维护。**本 PR 暂不加** `.git-blame-ignore-revs`——单维护者仓库，blame 通常追到 squash commit 后跳前一版即可。若后续真有反复溯源诉求再加。
- **commit 2 改 ci.yml 后第一次 push 触发 CI**：commit 1 (reformat) 推上去时 CI 会跑——此时新加的 `--check` 步骤还没合入，所以 commit 1 单独推应该过；commit 2 推上去（且分支已含 commit 1 的 reformat）时 `--check` 才生效，应该过。

### 4.2 回退

- 若仅 reformat 引发问题：`git revert <commit 1>` 一刀切回
- 若 CI 接入有问题：`git revert <commit 2>`，保留 reformat baseline

两者独立，互不耦合。
